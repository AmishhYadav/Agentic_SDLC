<objective>
Grow `ingest_config` from Phase 1's trivial passthrough into the real, blocking
CONN-01/02/03 entry point: load `.env` config (ADO_ORG/ADO_PROJECT/ADO_PAT,
GITHUB_REPO, REPO_MODE), run an ordered ADO PAT smoke-test (project access,
auth validity, write scope, best-effort expiry), and block the run with a
detailed per-check reason on failure — surfaced to the lead through the
existing status-derivation route, not swallowed.

Purpose: CONN-03's "immediately see a clear pass/fail" requirement (D-03: displayed with detail — scope, expiry, project access, never an opaque "run blocked") is the
first thing a lead experiences on every run. Getting this right first means
every later plan (team roster, docs fetch, plan generation) builds on a graph
that already knows how to fail loudly and specifically.
Output: `ado_client.run_smoke_test()`, a grown `ingest_config` node that calls
it and can halt the graph, `RunState` fields for the smoke-test result and
`repo_mode`, `runs.py` surfacing the smoke-test detail to the frontend, a
pytest scaffold (this is the first test file in the backend), and updated
`.env.example`/`requirements.txt`.
</objective>

    Returns `{"check": "project_access", "passed": bool, "reason": str | None}`.
    Non-JSON/203 response -> reason "PAT invalid or expired (non-JSON/203 response)".
    401 -> "PAT auth rejected (401)". 403 -> "PAT lacks access to this project (403)".
    Any other non-200 -> f"unexpected status {status_code}". 200 -> passed=True.

    `create_work_item("Task", {"System.Title": "ADO smoke-test — safe to delete", "System.AssignedTo": lead_email}, parent_id=None)`

    `{"check": "write_scope", "passed": bool, "reason": str | None}`; on
    RuntimeError, reason=str(exc); on success, passed=True, reason=None.
  - `async def check_expiry_best_effort() -> dict` — GETs
    `https://vssps.dev.azure.com/{org}/_apis/tokens/pats?api-version=7.1-preview.1`
    using `_auth_header()`. This is best-effort per Assumption A1/Open Question
    1 — never fails the overall smoke-test. On a 200 JSON response, extract
    `validTo` from the matching PAT entry if findable (best-effort text match
    is fine; do not over-engineer identifying "which" PAT). Returns
    `{"check": "expiry", "passed": True, "reason": <validTo string or "unknown (best-effort check unavailable)">}`.
    On any non-200/non-JSON response, return passed=True (does not block),
    reason="unknown (best-effort check unavailable)".
  - `async def run_smoke_test(lead_email: str) -> dict` — runs
    `check_project_access()` first; if it fails, short-circuit and return
    immediately (write-scope check is meaningless without project access) with
    shape `{"passed": False, "checks": [project_access_result]}`. Otherwise run
    `check_write_scope(lead_email)` and `check_expiry_best_effort()`, and
    return `{"passed": <project_access.passed and write_scope.passed>, "checks": [project_access_result, write_scope_result, expiry_result]}`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Lead's `.env` -> ado_client | ADO_PAT read from local file, never from network input; trusted within this single-lead MVP |
| ado_client -> Azure DevOps REST API | Outbound HTTPS calls carrying the PAT in a Basic auth header; ADO's TLS is the transport trust boundary |
| FastAPI response -> React frontend | smoke_test detail (reasons, no PAT value) crosses this boundary; must never include the raw PAT/token value |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | ado_client smoke-test functions | mitigate | Smoke-test `reason` strings never interpolate `os.environ.get("ADO_PAT")` directly — only HTTP status codes and generic phrasing; verified by code review (no `ADO_PAT` variable referenced inside any `reason=` string literal) |
| T-02-02 | Tampering | `_apis/tokens/pats` best-effort expiry probe | accept | Read-only GET, no write; failure is swallowed into "unknown" per design, cannot corrupt state |
| T-02-03 | Denial of Service | `check_write_scope` creates a real ADO work item on every run start | accept | Low-value target (single local lead, own ADO project); acceptable work-item clutter for a 2-day MVP, not a security risk |
</threat_model>
