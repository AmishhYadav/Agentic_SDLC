ADO REST call shapes (from 01-RESEARCH.md "ADO REST Call Reference" — authoritative for THIS plan):

Create epic: `POST https://dev.azure.com/{org}/{project}/_apis/wit/workitems/$Epic?api-version=7.1`,
header `Content-Type: application/json-patch+json`, body is a JSON array of
`{"op": "add", "path": "/fields/System.Title", "value": ...}` style ops.

Create task with parent link in one call: same endpoint with `$Task`, ops include
`/fields/System.Title`, `/fields/System.Description`, `/fields/System.AssignedTo`,
`/fields/Microsoft.VSTS.Scheduling.OriginalEstimate`, plus one op
`{"op": "add", "path": "/relations/-", "value": {"rel": "System.LinkTypes.Hierarchy-Reverse", "url": "https://dev.azure.com/{org}/{project}/_apis/wit/workItems/{epicId}", "attributes": {"comment": "linking to parent epic"}}}`.
`Hierarchy-Reverse` goes on the CHILD (task), pointing up to the parent (epic) — do not reverse this.

Read-back verify: `GET https://dev.azure.com/{org}/{project}/_apis/wit/workitems/{id}?$expand=relations&api-version=7.1`.
Check `fields["System.AssignedTo"]` resolved to the expected identity (not null/absent) and `relations`
contains an entry with `rel == "System.LinkTypes.Hierarchy-Reverse"` pointing at the parent's URL. Check
response `Content-Type` is JSON before parsing (a 203 HTML login page means the PAT is invalid/expired).

Auth header for every call: `Authorization: Basic {base64(":" + ADO_PAT)}` (empty username).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| FastAPI backend -> Azure DevOps REST API | Outbound HTTPS calls carrying the shared PAT in every request header; external service, org-controlled data written (real work items) |
| backend/.env -> ado_client.py / script_a | ADO_PAT, LEAD_EMAIL read from local environment into outbound request headers |
| stub_plan's suggested_assignee -> System.AssignedTo | Plan-controlled data (currently only ever the lead's own email per D-05) flows unvalidated into an ADO identity-resolution field |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-01-06 | Information Disclosure | ADO_PAT in outbound request headers / logs | mitigate | Never log full request headers in ado_client.py or script_a; PAT stays in-process, read once from os.environ, never echoed back in API responses to the frontend |
| T-01-07 | Tampering | ADO PAT scope broader than needed | mitigate | user_setup instructs "Work Items (Read & Write)" scope only, not "Full access" (research Pitfall 6) — documented in the checkpoint task's how-to-verify |
| T-01-08 | Spoofing | Expired/invalid PAT returns 203 HTML instead of clean 401 | mitigate | verify_work_item and create_work_item check response Content-Type is JSON before parsing; a non-JSON response is treated as a create_failed/verification failure, not silently parsed as empty JSON |
| T-01-09 | Repudiation / Denial of Service | Unbounded retry or parallel push on failure | accept | Phase 1's stub plan is only 1 epic + 2-3 tasks (D-05) — sequential push, no concurrency cap needed at this scale; explicitly deferred if later phases push larger plans (documented as a Phase 2+ concern, not re-litigated here) |
| T-01-10 | Information Disclosure | Real work item content (task titles/descriptions) pushed to a real ADO org | accept | This is the explicit purpose of Phase 1 — proving a real push works; content is the hardcoded, non-sensitive stub plan (D-05), not real project data |
</threat_model>
