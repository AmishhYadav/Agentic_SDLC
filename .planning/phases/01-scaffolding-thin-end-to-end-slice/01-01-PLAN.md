<objective>
Build the minimal straight-line LangGraph spine (`ingest_config -> stub_plan -> human_review -> push_to_ado`) wired into FastAPI with a durable file-backed `AsyncSqliteSaver` checkpointer, proving the interrupt/resume + restart-survival integration point (ORCH-01 partial per D-02, ORCH-02) before any real ADO logic exists. `push_to_ado` in this plan is a deliberate no-op stub (returns an empty "not yet implemented" report and sets `pushed=True`) so the full graph compiles and runs end-to-end; Plan 01-02 replaces this stub with the real ADO push.

Purpose: De-risk the two hardest LangGraph integration points (side-effect-free interrupt node, durable checkpointing across process restart) in isolation from ADO's own integration risk, so failures in one system don't mask failures in the other.
Output: A running FastAPI backend where `POST /runs` starts a run that pauses at `awaiting_review`, survives a backend restart, and `POST /runs/{id}/resume` unblocks it through to a (stubbed) completion state.
</objective>

```python
from pydantic import BaseModel
from typing import Literal

class Task(BaseModel):
    id: str
    title: str
    description: str
    suggested_assignee: str      # email/UPN string; ADO resolves it server-side
    estimate_hours: float
    skill_tag: str | None = None       # Phase 1: unused/None; Phase 2+ populates
    depends_on: list[str] = []          # display-only, no enforcement

class Epic(BaseModel):
    id: str
    title: str
    description: str
    tasks: list[Task]

class Plan(BaseModel):
    epics: list[Epic]

class PushResultItem(BaseModel):
    item_id: str
    ado_work_item_id: int | None = None
    status: Literal["created", "assignment_unresolved", "create_failed", "link_failed", "not_implemented"]
    detail: str | None = None

class PushReport(BaseModel):
    items: list[PushResultItem]
    all_succeeded: bool
```

`RunState` shape (backend/app/graph/state.py) — TypedDict, checkpointed to SQLite on every node transition:

```python
from typing import TypedDict
from app.models.plan import Plan, PushReport

class RunState(TypedDict, total=False):
    run_id: str
    lead_email: str          # used for self-assignment in Plan 01-02
    plan: Plan
    approved: bool
    pushed: bool
    push_report: PushReport
```

FastAPI route contract (backend/app/routers/runs.py) that Plan 01-03 (frontend) consumes directly:

- `POST /runs` -> `{"run_id": str, "status": "awaiting_review" | "running"}` (status reflects state immediately
  after `ainvoke` returns, which for this spine is always "awaiting_review" since the graph pauses at
  `human_review` on the first invocation)
- `GET /runs/{run_id}` -> `{"run_id": str, "status": "running" | "awaiting_review" | "completed" | "not_found",
  "plan": Plan | null, "push_report": PushReport | null}` — status is derived from
  `graph.aget_state(config)`: pending interrupt task present => "awaiting_review"; no pending task and
  `pushed` true => "completed"; otherwise "running". A restart-then-query of the same `run_id` MUST return
  the same status it had before the restart (this is the ORCH-02 proof).
- `POST /runs/{run_id}/resume` with body `{"approved": bool}` -> re-invokes
  `Command(resume={"approved": bool})` against the same `thread_id=run_id`, returns the same shape as
  `GET /runs/{run_id}` reflecting the post-resume state.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> FastAPI run-control endpoints | Unauthenticated local HTTP; explicit scoped MVP decision (CLAUDE.md: "No user auth. Single shared session, opened only by the lead"), not an oversight |
| .env file -> process environment | Local filesystem read of secrets (ADO_PAT, future ANTHROPIC_API_KEY) into process memory |
| FastAPI process -> SQLite checkpoint file | Local file writes containing full run state (stub plan content, lead email) — no PII beyond the lead's own email in this phase |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-01-01 | Information Disclosure | .env file | mitigate | .gitignore excludes .env (Task 1) before any commit; backend/.env.example contains only empty placeholders, never real values |
| T-01-02 | Tampering | POST /runs, POST /runs/{id}/resume (no auth) | accept | Explicit MVP scope: no user auth, single shared local session opened only by the lead (CLAUDE.md non-negotiable constraint) — not a bug, documented accepted risk for a local-only 2-day tool with no network exposure beyond localhost |
| T-01-03 | Denial of Service | SQLite checkpoint file (single-writer lock contention) | accept | Single local lead, one run at a time in practice for this phase; AsyncSqliteSaver's built-in lock is explicitly not for production concurrency per research Pattern 1 trade-off note, acceptable here |
| T-01-04 | Information Disclosure | RunState checkpointed to disk (lead_email, stub plan content) | accept | No sensitive data beyond the lead's own email in Phase 1's stub; local file only, not transmitted externally except in Plan 01-02's ADO push |
| T-01-05 | Repudiation | No audit log of who clicked Approve | accept | Single-lead local tool with no auth boundary to attribute actions across — out of scope per CLAUDE.md |
</threat_model>
